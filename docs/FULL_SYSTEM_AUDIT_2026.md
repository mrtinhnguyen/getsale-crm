## SECTION 1 — Architecture Review

### 1.1 Backend

- **Границы сервисов:** Чёткие по доменам (auth, user, crm, pipeline, messaging, bd-accounts, automation, analytics, team, websocket, ai, campaign). **Слабое место:** общая БД (один PostgreSQL) для многих сервисов — границы данных размыты; любой сервис может читать/писать таблицы другого (например campaign-service и messaging-service к `conversations`, `messages`). **Риск:** при росте — конфликты миграций, отсутствие контрактов схемы.
- **Домен и консистентность:** Бизнес-логика в роутах, отдельного domain-слоя нет. Транзакции ручные (BEGIN/COMMIT/ROLLBACK) в критичных местах (например createDealFromLead в crm-service). Не везде многозначные операции обёрнуты в транзакции — **риск целостности** при сбоях.
- **Связность:** Высокая через общую БД; межсервисные вызовы по HTTP (gateway → backend, messaging → bd-accounts). RabbitMQ для событий. Идемпотентность сделки по lead_id есть (409); в других местах идемпотентность не унифицирована.
- **Оценка:** **High** — общая БД без чёткого ownership схемы; **Medium** — не везде транзакции и идемпотентность.

### 1.2 Database

- **Нормализация:** В целом адекватная; есть связи org, FKs, отдельные таблицы для истории (stage_history, lead_activity_log).
- **Индексы:** Есть performance-миграции (CONCURRENTLY), составные индексы под запросы (messages, conversations, leads, campaign_participants). **Хорошо.**
- **Масштабирование:** Один инстанс PostgreSQL, пул max 8 (рекомендация PgBouncer в проде). При 10k conversations и 1M messages рост таблицы `messages` и тяжёлые срезы по дате/каналу — **риск**. Нет явной партиционирования/архивов.
- **Запись/чтение:** Смешанные; тяжёлые списки (чаты, сообщения) с пагинацией. **Medium** — при росте понадобится стратегия загрузки истории (lazy, окна, возможно read replicas).

### 1.3 Observability

- **Логирование:** @getsale/logger, requestLogger, структурированные поля. Correlation ID прокидывается. **Хорошо для старта.**
- **Метрики:** Prometheus (prom-client), /metrics, нормализация путей. Сервис-специфичные метрики (например deal_created_total). **Достаточно для MVP.**
- **Алерты:** В коде не заданы — **пробел для прода.** Нет SLA/базовых порогов (latency, error rate, queue depth).
- **Трейсинг:** Correlation ID есть; полноценного distributed tracing (Jaeger/OpenTelemetry) в коде нет — **Low/Medium** для масштаба.

### 1.4 Error handling

- **Откат транзакций:** В реализованных транзакциях (например crm-service deals) ROLLBACK есть. Не везде многшаговые операции в одном сервисе обёрнуты в транзакцию — **риск.**
- **Внешние API:** Вызовы к AI (OpenAI), BD Accounts (Telegram/GramJS). Retry есть в bd-accounts (GramJS reconnect, retryDelay) и в websocket (Redis retryStrategy). В messaging-service при вызове ai-service и bd-accounts **нет** явного retry/circuit breaker — **High** при нестабильной сети или перегрузке AI.
- **Идемпотентность:** Частично (сделка по lead_id, automation_executions). Универсальных idempotency keys для HTTP API нет — **Medium.**

**Итог Section 1:** Критично: общая БД без контрактов и риск при отказе внешних API без retry/circuit breaker. Высоко: не везде транзакции; средний риск по масштабированию БД и алертам.

## SECTION 2 — Frontend & UX Architecture

- **State:** Zustand по доменам (auth, contacts, leads, deals, messaging-*). Контексты для WebSocket, toast, theme, i18n. Масштабируемость Zustand достаточна; риск — разрозненные сторы без единого контракта (например нормализованные сущности).
- **UI-архитектура:** Next.js 14 App Router, dashboard layout, sidebar. Масштабируемость страниц нормальная; дублирование логики между страницами (загрузка, ошибки) — **Medium** (риск дублирования).
- **Правая панель (вкладки):** Жёстко заданы `ai_assistant` и `lead_card` в RightWorkspacePanel; нет реестра/плагинов. Добавление вкладки = правки типа, массива TABS и стора. **Расширяемость — Low** по сравнению с Telegram Web / Linear / Notion.
- **Производительность:** react-virtuoso для длинных списков сообщений; есть LRU для медиа. Риск тяжёлых ре-рендеров при обновлении больших сторов без селекторов — **Medium**, требуется проверка под нагрузкой.
- **Сравнение с эталонами:** Ближе к «одна основная панель + фиксированный набор вкладок»; до многопанельного workspace (как в Notion/Linear) далеко. Для AI-расширения текущая панель достаточна для одного ассистента; мульти-инструменты/виджеты потребуют рефакторинга панели.

**Итог Section 2:** Сильные стороны: понятный стек, виртуализация, WebSocket. Слабости: слабая расширяемость правой панели, возможное дублирование и риски производительности при росте данных.

## SECTION 3 — AI Assistant Architecture

- **Реализация:** AI реализован как **набор фич** (summarize, analyze, drafts), а не как единый слой «интеллекта» с общим контекстом и версионированием моделей.
- **Структурированные выходы:** Analyze возвращает JSON (project_summary, stage, risk_zone, recommendations, draft_message). Версионирование промптов: PROMPT_VERSION в коде; версионирование ответов в БД ограничено (model_version в conversation_ai_insights).
- **Сохранение инсайтов:** Есть таблица `conversation_ai_insights`; чат AI-ассистента в localStorage по conversationId — не синхронизируется между устройствами и не используется для обучения/аналитики.
- **Токены:** Учёт по запросам (rate limit 200/час на org), **не по токенам** — при переходе на дорогие модели или большие контексты стоимость непредсказуема.
- **Смена модели:** Модели задаются env (AI_MODEL_DRAFT/ANALYZE/SUMMARIZE); смена возможна, но без A/B и без учёта стоимости по моделям.
- **Разделение промптов:** Один файл prompts/index.ts; разделение по фичам есть, но нет шаблонизации/инжекции контекста из продукта (воронки, полей CRM) — **средний** риск сцепления с продуктом.
- **Риски:** Связь с messaging (вызовы AI из messaging-service); при падении AI часть сценариев ломается. Утечки данных: в промпты уходит до 12k символов переписки — нужна политика хранения и доступа.

**Итог Section 3:** AI — уровень «feature set», не «intelligence layer». Для конкуренции с Notion AI / Intercom нужно: учёт токенов, единый контекст, расширяемая панель и, при необходимости, кэш/дедупликация запросов.

## SECTION 4 — Product Consistency Audit

- **Терминология:** «Лид», «сделка», «воронка», «conversation», «чат» — в целом согласованы; в коде и UI есть «BD accounts», «chats», «conversations» — стоит зафиксировать глоссарий в одном месте.
- **Жизненный цикл:** Lead (pipeline/stage) → deal (при создании из лида); conversation с полями won_at, lost_at, first_manager_reply_at. Логика связная; этапы воронки настраиваемые.
- **Модель лида:** Лид привязан к контакту, воронке, стадии; возможен переход в сделку. Модель пригодна для роста при сохранении правил автоматизации и истории.
- **RBAC:** Роли (owner, admin, supervisor, bidi, viewer), role_permissions для messaging и bd_accounts. Выравнивание с продуктом есть; для enterprise не хватает детализации (например по полям/воронкам).
- **Скрытые ограничения:** Жёсткая привязка правой панели к двум вкладкам; общая БД ограничивает независимое масштабирование сервисов; отсутствие мульти-региона и изоляции по данным (например по юрисдикции).

**Итог Section 4:** Продукт в целом согласован; основная техническая граница — общая БД и монолитность панели.

## SECTION 5 — Scalability & SaaS Readiness

- **10k conversations, 100 concurrent users, 1M messages:**
  - **БД:** Рост таблиц `messages`, `conversations`, `lead_activity_log`; без партиционирования и политики архивации запросы по дате/фильтрам могут деградировать. **Первым упрётся** объём сообщений и тяжёлые аналитические запросы.
  - **Загрузка истории:** Пагинация вверх есть; при 1M сообщений важна стратегия «окон» и индексы — частично закрыто, но не проверено под нагрузкой.
  - **AI cost:** Фиксированный лимит запросов в час без учёта токенов — при росте пользователей и длинных контекстах расходы могут взрывно расти.
  - **Синхронизация:** Telegram sync через GramJS; при большом числе аккаунтов и чатов возможны узкие места на стороне bd-accounts-service и лимитов Telegram API.
  - **События:** RabbitMQ; при росте нужна метрика глубины очередей и DLQ (по ROADMAP — после Observability). Сейчас — **пробел.**
  - **Фоновые задачи:** Campaign отправки по расписанию, генерация черновиков по событиям — выполняются в сервисах; отдельной очереди джобов с приоритетами и повторными попытками нет — **риск при пиках.**

**Итог Section 5:** Система выдержит рост до определённого порога; первыми точками отказа — объём `messages`, нагрузка на AI и синхронизацию Telegram, отсутствие DLQ и явной стратегии фоновых джобов.

## SECTION 6 — Governance & Long-Term Maintainability

- **Документация:** После очистки 2026 — единый источник по состоянию (STATE_AND_ROADMAP), архитектура (ARCHITECTURE), запуск (GETTING_STARTED, TESTING). Требуется дисциплина обновления.
- **Архитектура:** Явной политики «заморозки» архитектуры или обязательного ревью изменений схемы/сервисов нет.
- **Дисциплина изменений:** Миграции Knex есть; нет явного процесса (например обязательный прогон миграций в CI, защита от прямых изменений в прод-БД).
- **Тесты:** Vitest в корне, crm-service, pipeline-service; e2e stage3-e2e. Покрытие критичных сценариев не полное; много сервисов без юнит/интеграционных тестов — **риск регрессий.**
- **Миграции:** Есть папка и скрипты; готовность к откату (down-миграции) и к длительным миграциям (online schema change) не унифицирована.

**Итог Section 6:** Требуется целенаправленная заморозка ключевых контрактов и дисциплина обновления одного набора документов.

## SECTION 7 — Competitive Benchmarking

- **HubSpot / Pipedrive / Close:** Зрелые CRM с глубокой аналитикой, автоматизациями и интеграциями. Getsale сильнее в **messaging-first** и **Telegram-native**; слабее в широте интеграций, отчётах и зрелости автоматизаций.
- **Telegram-based sales:** Позиционирование сильное — единый inbox, папки, лиды из чатов, кампании. Не хватает массовых сценариев (шаблоны, вариации, A/B) и глубокой аналитики по каналам.
- **AI-native CRM:** По сравнению с появляющимися продуктами — AI пока «дополнение» (суммаризация, черновики), а не ядро принятия решений и прогнозов. Нет предсказания конверсии, авто-тегов, скоринга лидов на базе AI.

**Итог Section 7:** Архитектурно система способна конкурировать в нише «Telegram-first sales CRM»; для выхода в широкий рынок нужны усиление аналитики, AI как слой решений и стабильность при масштабе.

## SECTION 8 — Final Verdict

1. **Overall maturity score: 5.5–6 / 10** — рабочая MVP-система с сильной стороной в мессенджинге и воронке; ограничения по масштабу, наблюдаемости и AI.
2. **Architecture maturity: Startup → Growth** — ещё не «Scale-ready»: общая БД, нет DLQ, нет стратегии партиционирования/архивов, слабый retry/circuit breaker на внешних вызовах.
3. **AI maturity: Feature-level** — полезные фичи (summarize, analyze, drafts); нет слоя интеллекта, учёта токенов и стратегии моделей.
4. **Top 5 critical risks:**
   - Общая БД без явного ownership и контрактов схемы между сервисами.
   - Отсутствие retry/circuit breaker при вызовах AI и bd-accounts из messaging (и других) — отказы внешних API бьют по UX.
   - Рост таблицы `messages` и тяжёлые запросы без партиционирования и стратегии архивации.
   - Отсутствие алертов и DLQ — сбои и переполнение очередей обнаруживаются поздно.
   - Документация и тесты не успевают за продуктом — риск регрессий и онбординга.
5. **Top 5 strategic strengths:**
   - Чёткое разделение сервисов по доменам и событийная модель (RabbitMQ).
   - Correlation ID, структурированные логи, базовые метрики Prometheus — хорошая база для observability.
   - Идемпотентность критичных операций (сделка из лида, частично automation).
   - Telegram-first UX и связка воронка–чаты–лиды–сделки — сильное продуктовое отличие.
   - Использование PostgreSQL, Knex, миграций и индексов — предсказуемая эволюция схемы.
6. **Recommended next strategic direction:**
   - **Краткосрочно:** Reliability — retry/circuit breaker для внешних вызовов, алерты по метрикам и здоровью очередей, DLQ после Observability.
   - **Среднесрочно:** Scale infrastructure — стратегия партиционирования/архивов для `messages`, проверка нагрузки на 10k conversations, при необходимости read replicas.
   - **Параллельно:** Product refinement — унификация документации (выполнена в 2026), глоссарий, приоритизация недостающих CRUD/валидаций по STATE_AND_ROADMAP.
   - **Далее:** AI expansion — учёт токенов, возможность смены моделей и A/B, сохранение контекста AI не только в localStorage.
